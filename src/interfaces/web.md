# Web Interface

The web interface is MRA's main product. Students land here from Travis's sales page, sign up, and spend their actual working hours inside it. Because it's a work tool used by paying adults on real campaigns, it's designed desktop-first with mobile as a fully responsive secondary surface. Dark-only (Asphalt canvas with grain). Never white background.

~~~
Web framework: Vite + React. State management: Zustand for the session-level store (current user, projects, concepts cache, artifacts). SWR is not used — the app is small enough that we load relevant data on session start and mutate optimistically.

Set `"defaultPreviewMode": "desktop"` in `web.json`. This is a desktop-first product.
~~~

  - `CHAT` (all conversations, most recent first)
  - `WORKFLOWS` (the catalog of available workflows, with unlocked/locked state)
  - Admin-only: `ADMIN` at the bottom, Mojo Red accent dot
  - User's name + plan (`FULL ACCESS` or `FREE`) pinned at the bottom
  - Collapses to a 56px icon rail on mobile

- **Main content (Asphalt with grain):** Whatever page the user is on.

The left rail nav items use `Label` style for the section header (`PROJECTS`), and `UI Body Small` for the individual items below. The active item has a Mojo Red left-border (2px) and slightly brighter text.

~~~

~~~

### Verification code input

This is the first impression and must feel premium. Six individual Gunmetal boxes, 56px square, Switzer 24/600, Bone White text, `1px solid Graphite` border. Active box has a `2px solid Mojo Red` inner border (not outer — outer shifts layout).

- Typing a digit auto-advances to the next box.
- Backspace on an empty box jumps to the previous box.
- Pasting a 6-digit string fills all boxes and auto-submits.
- On success: subtle red-to-success transition, then the whole form fades out as the app shell fades in. No page reload.
- On error: the boxes shake once (4px translate), error message appears in Rust below the boxes in `Caption` style. The boxes never resize or reposition. A `Resend code` button is always visible below with a 30-second cooldown displayed as `Resend in 23s` in Mono Detail.

~~~
Implementation notes:
- Use the `@mindstudio-ai/interface` auth SDK (`auth.sendEmailCode`, `auth.verifyEmailCode`).
- Test emails (`remy@mindstudio.ai`, `555` phones) work with code `123456` — ensure a test path during development.
- Layout must not shift between "enter email" and "enter code" states. Use opacity transitions on the code input container, not conditional mount.
- After `verifyEmailCode` succeeds, `onAuthStateChanged` fires, the app redirects to `/dashboard`.
~~~

## Mr. A Chat

The conversational surface. Can be accessed project-scoped (from inside a project's Chat tab) or global (from the left rail's `CHAT` section — a thread not tied to any project).

**Layout:** Single scrollable column centered on the canvas, max-width 720px. Composer fixed at the bottom.

**Thread list (left drawer, collapsible):** When inside the Chat surface, a second rail slides in between the main left rail and the content, showing the user's threads for the current scope (global or project-scoped). Active thread is highlighted. `+ NEW THREAD` button at the top.

**Empty thread state (when a new thread is opened with no messages yet):**
- Editorial Headline greeting in Montagu Slab — something brand-grounded, e.g. "What are we working on?"
- Below: a list of 3-5 suggested starter prompts as Ironwood chips. Rotating selection drawn from recent common asks (e.g. "Help me sharpen my niche," "Draft a T1 email for [partner]," "Review this sales page," "Walk me through a coffee date"). Clicking a chip sends it as the first message.
- A Prose note at the bottom: "Ask me anything — I've got all of Travis's stuff organized and cited."

**Active thread:**
- User messages and Mr. A replies alternate, with 40px vertical gap between consecutive items from different speakers.
- User messages follow the pattern in `@brand/references/chat-pattern.html`: right-aligned, Gunmetal, 75% max-width, 14/18 padding, `UI Body` type, Graphite border, 4px radius.
- Mr. A replies follow the canonical pattern: 32px Mojo Red M monogram + `MR. A` label meta row, then Prose text indented 42px. No bubble, no "responding as Travis" subtitle.
- Citation chips appear below the reply, pre-allocated container, fading in after stream completes.

**Streaming behavior:**
- When the user sends a message, it appears immediately in the thread.
- The M monogram meta row appears right after, with an animated three-dot "thinking" indicator in the Prose line (Dust color, three dots that pulse).
- Tokens stream into the Prose area as they arrive. A subtle red vertical cursor blinks at the end of the current streaming text.
- Tool calls (searching concepts, fetching sources) appear as small Mono Detail status lines between thinking and streaming: `SEARCHING · coffee dates / cold outreach`, `FOUND · 4 sources`. These stay in place once streaming starts and fade to a collapsed `4 SOURCES CITED` line after completion.
- Citation chips fade in below the message after `done` event.

~~~
Technical:
- Use `createAgentChatClient()` from `@mindstudio-ai/interface`.
- Render markdown in replies using `streamdown` (handles unterminated blocks during streaming).
- `use-stick-to-bottom` for scroll behavior.
- `react-textarea-autosize` for the composer.
- Throttle `onText` render to ~60ms batches to smooth token delivery.
- The Prose container needs `min-height: 1.55em` reserved to prevent the composer hopping.
- Keep a tool-activity buffer during streaming so the `FOUND · N sources` summary renders correctly.
- Tool call events are `onToolCallStart` / `onToolCallResult` — render them as inline status lines in Mono Detail.
- On `onError`, show a Rust-colored inline error in place of the message with a `Retry` action.
~~~

**Composer:**
- Fixed at the bottom of the chat surface.
- Gunmetal background, Graphite 1px border, 4px radius, 16px padding.
- Auto-growing textarea (1 line to max ~8 lines, then scrolls internally).
- Placeholder: `Ask Mr. A anything...`
- Submit button: Mojo Red filled arrow icon, right-aligned, disabled when input empty or streaming.
- During streaming, the submit button becomes a `STOP` button (outlined Mojo Red) that aborts the stream via `AbortablePromise.abort()`.
- Attachment affordance (paperclip icon, Dust color) for future file uploads — hidden for v1 or visible as disabled tooltip "Coming soon."

**Thread auto-titling:** After the first exchange completes, the thread title is auto-generated and replaces the placeholder. Uses the platform's built-in thread title generation.

### Voice input and output

The mobile-out-and-about use case matters. A student pulls over, opens MRA, and wants to dictate what a partner just said on the phone, then hear Mr. A's response back without looking at the screen. Voice is a first-class input method in chat, not an afterthought.

**Voice input (dictation):**

A mic button sits in the composer next to the send button. Tap to start recording, tap again to stop (single-tap toggle, not push-and-hold — push-and-hold is awkward when the student is driving or walking).

- **While recording:** the composer's placeholder is replaced with a live audio waveform animation in Mojo Red. A subtle `Listening...` label in Dust (Mono Detail style) sits below the waveform. The mic button turns into a Mojo Red filled square (stop icon).
- **On stop:** the audio is sent to `mindstudio.transcribeAudio()` in the background. The composer shows a brief Mono Detail `Transcribing...` state with a stationary Graphite progress block. When transcription returns, the text populates the composer (editable) and the send button lights up. The student can send as-is or edit first.
- **Error handling:** if transcription fails or is empty, show a Rust-colored inline message below the composer: `Couldn't hear that. Try again.` The audio is discarded; composer returns to normal state.
- **Permissions:** browser mic permission is requested on first use. If denied, the mic button shows a small lock icon with a tooltip linking to permission settings.

**Voice output (playback):**

Each of Mr. A's replies has a small speaker icon tucked at the bottom-right of the message (Dust color, subtle — not competing with citation chips). Tap to play the reply back as synthesized speech. Tap again to pause.

- **Playback state:** while playing, the speaker icon becomes a Mojo Red pulsing ring. A thin Mojo Red progress line runs underneath the message body tracking playback position. Tapping anywhere on the message pauses.
- **Auto-play toggle:** a setting in the account menu (`Auto-play Mr. A replies`) that, when enabled, automatically plays each reply as it completes streaming. Default off. When enabled, a small `AUTO-PLAY ON` Label indicator sits in the chat header so it's not forgotten.
- **Voice selection:** we use a single consistent voice for Mr. A — warm-neutral, medium-paced, no regional accent. Not an affected Travis impression. Selected during build from `mindstudio.textToSpeech()` available voices; the spec doesn't lock in a specific voice ID but directs toward something grounded and focused (not overly smooth or smarmy, not robotic). The same voice is used everywhere Mr. A speaks.
- **Handling citations and markdown in speech:** when TTS plays a reply, citation markers like `[1]` and `[2]` are skipped (not spoken). Markdown italics, code blocks, and links are stripped to plain text before synthesis.

**Mobile-specific affordances:**

On mobile, the voice experience is the primary interaction mode for many sessions. The mic button is enlarged to ~48px tap target (desktop is smaller). A secondary affordance: on the chat surface's empty-state screen, a prominent `START TALKING` button appears below the suggested prompts on mobile only, opening straight into recording mode so the student doesn't have to hunt for the mic.

~~~
Technical implementation:
- Voice input uses the browser's `MediaRecorder` API to capture audio (WebM/Opus on most browsers, MP4/AAC on Safari). After recording, the audio blob is uploaded via `platform.uploadFile()` from `@mindstudio-ai/interface`, then the resulting CDN URL is passed to a backend method that calls `mindstudio.transcribeAudio({ audioUrl })`. The transcription is returned to the frontend and populated into the composer.
- Voice output uses `mindstudio.textToSpeech()`. The backend method synthesizes the reply's plain-text version (citations and markdown stripped) and returns an audio URL. The frontend plays it via an `HTMLAudioElement`. TTS is generated on-demand when the student taps the speaker — not pre-generated for every reply (saves cost, only pays for what's played).
- Look up the currently-recommended model for both `transcribeAudio` and `textToSpeech` via `askMindStudioSdk` during build; don't guess at model IDs.
- On iOS Safari, audio playback must be triggered by a user gesture — the first tap on the speaker icon works, but auto-play mode on iOS requires a one-time `Enable auto-play` prompt on first use, tied to a user gesture.
- Cache generated TTS audio per-message so replaying doesn't re-synthesize. Cache keyed on message ID.
- Waveform animation during recording is cosmetic — CSS-based animated bars are fine. No need to render actual audio levels.
~~~

- **Description**: the chunk's pre-written description in Prose, italicized, Dust color.

- **Action button** (sticky bottom): `OPEN AT 14:22 →` (not a generic `OPEN IN COURSE`). Reads the timestamp off the chunk and renders it explicitly so the student knows exactly where they're going. The button opens the chunk's `linkUrl` in a new tab. If the URL contains a timestamp fragment (as the ETL handoff doc recommends), the player jumps straight to the moment. If it doesn't, the student lands on the course page with Mr. A's prose giving them the timecode to seek to.


## Workflow walkthrough

The tactical workflow surface. Three-column layout on desktop.

### Left column — Step rail (~200px, Ironwood)

Vertical list of numbered steps in Label style:
```
01 · VALIDATE NICHE      ← Brass check (completed)
02 · RESEARCH PARTNERS   ← Mojo Red accent (current)
03 · ATTRACTION VIDEO    ← Smoke (upcoming)
04 · SKOOL GROUP         ← Smoke (upcoming)
05 · COLD OUTREACH       ← Smoke (upcoming)
06 · TRACK & ITERATE     ← Smoke (upcoming)
```

Clicking a completed step navigates back (non-destructive; the student can review or re-run that step). Clicking an upcoming step does nothing (they're locked until prior steps are done).

Top of rail: workflow title in Editorial Subhead (`COFFEE DATES + GIVING FUNNEL`) and a small `Label`-style line showing `2 OF 6 · UPDATED 3H AGO`.

Bottom of rail: a small ghost-outlined `Exit to project` link in Smoke.

### Center column — Canvas (flexible, Asphalt with grain)

- **Step title** at the top in Editorial Headline (Montagu Slab 32/600). e.g. "Research your partner list."
- **Step intro** in Prose — 2-4 paragraphs of Travis coaching for this step. Draws from the concept library. Citation chips at the end of the intro in a row (Mono Detail).
- **Step inputs** — whatever the student needs to provide. For step 2 (research partners): the niche doc is already known; this step just needs a confirm-and-run. For step 3 (video): the student picks a format (talking head vs. Loom screenshare) and a length target.
- **Action area (sticky at bottom):** `BACK` (outlined Bone White) and `CONTINUE` / `GENERATE` (Mojo Red filled) buttons. Progress is saved on every step.

### Right column — Artifact panel (~40% on desktop, Paper surface with grain)

- **Artifact header:** Label-style title in Coal (`YOUR NICHE DOC` or `YOUR PROSPECT LIST`). Small Mono Detail metadata: `DRAFT · v3 · LAST EDITED 2M AGO`. A `Copy` button (outlined Coal) and a `Download` button for text/PDF export.
- **Artifact body:** The content, editable inline. For text artifacts, a rich text editor. For structured artifacts (prospect list), an editable table.
- **Coaching footnote:** Under the artifact, a Paper-surface `Newsprint` rule, then a small italic Prose-style footnote in Coal (darker) quoting the most relevant Travis moment for this artifact, with a citation.

On mobile, the artifact panel stacks below the canvas. The student completes the canvas first, then the artifact populates and they scroll to it.

### Generation flow (step-by-step behavior)
Each step is a method call that runs a task agent, followed by an adversarial review pass:

1. Student clicks `GENERATE` or `CONTINUE`.
2. The canvas briefly shifts to a streaming state: Mr. A coaching prose streams in explaining what's happening, followed by inline Mono Detail status lines as tools run (`SEARCHING CONCEPT: niche-selection`, `FETCHING 3 SOURCES`, `DRAFTING`). The artifact panel shows a stationary Graphite skeleton (no shimmer) reserving space for the output.
3. When generation completes, a `REVIEWING` status line appears briefly as the reviewer agent checks the artifact.
4. Based on the review verdict:
   - **Pass** — artifact fades into the panel, continue button activates.
   - **Revise** — status flips to `REFINING` and the artifact regenerates silently. Student sees slightly longer generation time.
   - **Surface issues** — artifact fades in as usual, but a small Rust-colored `Label`-style tag appears at the top of the artifact header: `MR. A FLAGGED 2 CONCERNS`. Tapping it expands a section above the artifact body with the issues in Prose (Coal on Paper), each with a suggested revision the student can accept inline.
5. Student can edit the artifact inline, `REGENERATE` (new generation + new review), or `CONTINUE` to the next step.

~~~
Backing stack: every workflow step is a method using `mindstudio.runTask()` for generation, followed by a second method using `runTask()` for review. The review method uses a different model family when practical (if generation is Anthropic-class, review is OpenAI- or Google-class). Fire-and-forget from the frontend; the backend updates the `workflow_runs` record and the related `reviews` row. Frontend polls or streams to pick up state.

For simpler steps (short text outputs), use `stream()` instead of `runTask()` for lower latency, but still run the review pass. Use `runTask()` for complex steps (research, structured output).

The `MR. A FLAGGED CONCERNS` visual treatment is deliberately honest — it's better for students to see Mr. A questioning his own output than to silently hand them a questionable artifact. Over time, surfaced concerns teach students to edit artifacts critically rather than copy-paste.
~~~

### Resume behavior

When a student navigates back to a workflow run that's in-progress, the step rail shows their completed steps and drops them on the current step. The canvas and artifact panel restore from saved state. No data loss, no re-prompting.

### Content tab
The primary ingestion interface and the window into the library's quality over time. Has several sub-sections stacked vertically.

**Programmatic ingestion (the primary path).**
- A single `Label`-style line at the top of the tab: `POST /_/api/ingest/source` with a `Copy cURL` button and a link to the ETL handoff doc.
- Below it, a live job list: the last 20 ingestion jobs, one row each. Each row shows: job ID (Mono Detail), source (`API` / `Upload` / `Scenario`), started time, duration, total files, total chunks, status (running / completed / failed), and a `View details` action. Clicking expands to show per-file results and any errors.
- Status rows update in real-time via polling.

**Drag-and-drop (ad-hoc).**
- A simple drop zone below the job list: `Drop markdown files here to ingest`. Accepts single or batch upload.
- Uploaded files are POST-ed through the same pipeline as the API; the resulting job appears at the top of the job list.
- Useful for spot-fixing a single chunk, testing ETL changes, or small updates.

**Library stats.** A row of Tanker-set metrics: `12,847 SOURCES`, `14,293 CONCEPT LINKS`, `11 CONTEXTS`, `147 CANDIDATE CONCEPTS`, `47 KNOWLEDGE GAPS`. Tanker numbers, UI Body labels. Each is clickable:
- `CANDIDATE CONCEPTS` → opens the new-concept review queue (a modal with surfaced-but-not-yet-added concepts the linker proposed, each with evidence and an `Add to ontology` / `Dismiss` action).
- `KNOWLEDGE GAPS` → opens the gaps panel (described below).

**Knowledge Gaps panel.**
This is where Mr. A's honesty becomes actionable. Every time Mr. A or a workflow agent couldn't ground an answer in the library, a row appears here. The panel is a clustered list, grouped by normalized-tag (e.g. "cold-email-saas: 13 students," "refund-handling-post-auction: 4 students"). Each cluster shows:
- The normalized tag in Editorial Subhead
- A count of how many students asked about this
- A sample verbatim question (Mono Detail, truncated)
- A `Resolve` action — opens a text input where the admin can note the resolution ("added BEM-M07 covering this" or "documented as out-of-scope, won't cover") and mark the gap resolved.

Clusters are the default view. A `View individually` toggle expands to the raw list of questions if the admin wants to see specific phrasings. Gaps auto-resolve when a related concept or source is newly ingested (matched by the normalized-tag logic).

Over time this panel is the clearest signal of what Travis should record next, and what the ingestion pipeline is missing.

~~~
The Knowledge Gaps panel is a load-bearing feature, not a diagnostic afterthought. Admins should check it every week or two. The clustering is powered by a small task agent that tags gaps with short category labels; similar questions cluster together automatically.
~~~

### Ontology tab

See the full v1 ontology (4 North Stars, 16 Concepts, 16 Skills, 11 Contexts) and edit. Primary use cases:
- **Adding a Context.** Before ingesting content for a new course (e.g. BEAM), add the Context here with id, name, and aliases.
- **Completing empty descriptions.** The "Complete the ontology" queue surfaces rows with empty descriptions (COMMUNITY_AS_ASSET, OFFER_DOMINO, etc.) so the admin can fill them in over time.
- **Promoting candidate concepts.** New concepts surfaced during ingestion appear here with evidence and an action to promote them into the formal ontology.

Layout: editorial table similar to the Users tab. Four sub-rows for each ontology layer (North Stars, Concepts, Skills, Contexts), each editable inline. No drag-reordering for v1.

~~~
Admin table loads in a single batch (up to 500 users, no pagination in v1). Revocation is optimistic: row dims to Smoke immediately, rolls back if backend fails.

For programmatic ingestion at scale, the ETL posts to `/_/api/ingest/source` or `/_/api/ingest/batch` with the admin API key. A companion CLI (`npx mra-ingest <dir>`) walks a directory of markdown files and POSTs each file with retry and rate-limit handling. The CLI is not part of the MRA codebase for v1 — it's a thin wrapper script the admin can run locally. Docs live in `src/references/etl-handoff.md`.

## Responsive behavior

- Desktop (≥1024px): full three-column layouts, full left rail, full project dashboard table.
- Tablet (768-1023px): collapsible left rail, workflow still 2-column (canvas + artifact stacked below), chat single-column.
- Mobile (<768px): left rail is a full-height drawer, workflow single-column, project dashboard becomes a single-column list, chat single-column.

Mobile is a real use case — many students check in from their phone during the day to respond to a partner reply. The chat surface and the project detail page must be completely usable on mobile. The workflow walkthrough is acceptable but heavier on mobile; students will probably do most workflow work from desktop.

## Free tier specifics

A `free` user sees the full shell, but:
- `WORKFLOWS` in the left rail shows only `Validate Your Niche` as enabled. All others are Smoke-colored with a small `PAID ACCESS` badge.
- `CHAT` in the left rail shows as disabled with a tooltip: `Chat with Mr. A is paid access. Reach out to Travis's team for full access.`
- `PROJECTS` works normally, but projects are limited to containing only the Validate Your Niche workflow run and its niche doc artifact.
- A persistent strip at the bottom of every page (Ironwood, 48px) reads: `Unlock the full Mojo Results Accelerator →` with a link to Travis's sales page.

The lock is honest — free users can see what they're missing. What they have (a real, polished niche doc) is genuinely useful on its own and is a strong hook into the full product.
